════════════════════════════════════════════════════════════════════

   - BFAS/IPIP items → aspect scores → domain scores computed deterministically.

4. TRUST CONTRACT (provenance + disclosure)
   - Every user-facing personalization must cite which user-visible inputs were used.
   - Whitelisted signals only. Private taste profile is never shown to other users.

5. API KEY CONTRACT
   - GEMINI_API_KEY stays server-side. Never in client bundle, browser logs, or docs.
   - Use process.env.GEMINI_API_KEY in server code only.

SKILL 1 — kesher-personality-assessment
Progressive BFAS/IPIP Administration
════════════════════════════════════════════════════════════════════

INSTRUMENT SELECTION:
- Use: BFAS (100 items, public domain) or IPIP-NEO (public domain)
- Block: BFI-2, HEXACO-60, ECR-R (non-commercial / academic only)

SCORING:
- Extraversion = mean(Enthusiasm, Assertiveness)
- Agreeableness = mean(Compassion, Politeness)
- Conscientiousness = mean(Industriousness, Orderliness)
- Neuroticism = mean(Withdrawal, Volatility)
- Openness = mean(Intellect, Openness)
- Reverse-keyed: score = 6 - raw (for 5-point Likert)

ADMINISTRATION FLOW:
- Place in Settings/Insights — never first-run onboarding
- Show consent gate before first item
- Present in blocks of 10–15, allow save-and-resume
- After ~30 items, offer domain-only reflection; continue for aspect-level

QUALITY CHECKS (run before scoring):
- Straightlining: flag >10 consecutive identical responses
- Speed: flag median <1.5s per item
- Inconsistency: flag >2 SD variance in reversed-item pairs
- Instructed-response: flag any missed attention items
- Completion: flag <80% completion
- If flagged: show "reflection unavailable" — never force-score

REFLECTION CARDS:
- Route through Vertex AI with structured output
- Allowed: "Your responses suggest...", "You tend toward...", probabilistic language
- Forbidden: percentiles, raw scores, "You are...", compatibility predictions, clinical terms
- User-facing domain labels: Neuroticism→"Emotional Steadiness", Extraversion→"Social Energy",
  Openness→"Curiosity & Openness", Agreeableness→"Warmth & Consideration",
  Conscientiousness→"Planning & Follow-through"

BILINGUAL:
- Store items keyed by item_id + locale
- Score identically regardless of language
- Do NOT compare scores across languages until invariance established

SKILL 2 — kesher-consent-ux
Trust Hub, Grants Ledger & Section 11 Compliance
════════════════════════════════════════════════════════════════════

SECTION 11 NOTICE (required on every consent screen):
- Voluntariness: "This is optional. Declining does not affect your core Kesher experience."
- Purpose: specific purpose of this data collection
- Controller: Kesher entity name + contact
- Recipients: who receives data and why
- Refusal consequence: what happens if user says no
- Rights: access, correction, deletion rights exist
- AI disclosure: if AI processes data, disclose risks and possible adverse consequences

TRUST HUB (required settings screen):
- Assessment Status (Completed/In Progress/Not Started) + Reset button
- Active Shares: list of recipients, what they see, revoke button per share
- AI Usage toggle: "Use my personality for recommendation improvement" (default OFF)
- Data Controls: Export (JSON), Delete All, Reset Scores
- Consent History: timestamped log of all consent actions

GRANTS LEDGER (backend source of truth):
- Every share, AI usage, or cross-user reflection requires a ledger entry
- Revocation is immediate and cascading
- Ledger entries are immutable audit records

ANTI-DARK-PATTERN RULES:
- All sensitive toggles default OFF (never pre-checked)
- Neutral decline language: "Not now" (never confirm-shaming)
- Revocation effort ≤ granting effort (max 2 taps from Trust Hub)
- Separate consent per purpose (no bundled consent)
- Show consent prompt once, then only in Settings

SKILL 3 — kesher-permissioned-sharing
Share Cards, Mutual Consent & Revocation
════════════════════════════════════════════════════════════════════

CORE RULE: No personality information crosses user boundaries without explicit,
previewed, recipient-scoped consent. Nothing is sent automatically.

CARD TYPES:
- Basic Share Card: domain-level scores, 2–3 behavioral sentences per domain
- Deeper Share Card: domain + aspect scores, richer narrative, separate consent required
- Mutual Reflection Card: bilateral consent required, identical content to both users

SHARE FLOW:
1. User generates card → previews exact text → selects specific recipient → confirms
2. Recipient sees notification → taps to view → card shows "Shared by [Name] • Revocable anytime"
3. Sender can revoke anytime → card disappears from recipient

DEEPER CARD REQUEST:
- Request can only be sent once per pair
- No reminder or follow-up if ignored
- Sender's decision is never disclosed to requester

REVOCATION CASCADES:
- Revoke basic card → basic card removed from recipient; mutual reflection invalidated
- Reset assessment → all cards revoked for all recipients
- Delete account → all cards removed everywhere

NEVER INCLUDE IN CARDS:
- Raw item responses or specific quiz answers
- Numeric scores or percentiles
- Comparison to population norms
- Attachment labels (anxious, avoidant, secure)
- Clinical terminology
- Information from private taste profile or behavioral signals

SKILL 5 — kesher-privacy-recommendation
Three-Layer Recommendation Architecture
════════════════════════════════════════════════════════════════════

THREE LAYERS:
- Layer 1 (Silent personalization): explicit profile signals only. No personality.
- Layer 2 (Safe explanation): whitelisted public signals only. No personality.
- Layer 3 (Permissioned personality): consent-gated. BLOCKED until validation complete.

LAYER 1 ALLOWED SIGNALS: values, intent, observance/lifestyle, logistics, explicit preferences, behavioral feedback (likes/passes — NOT for personality inference)

LAYER 1 PROHIBITED: personality scores, private taste vectors, inferred traits, attachment labels, special-sensitivity signals

LAYER 3 PREREQUISITES (all must be met before enabling):
1. User completed assessment
2. User explicitly consented to AI recommendation use
3. Psychometric validation confirms incremental validity
4. Measurement invariance established
5. A/B testing shows personality adds value over explicit baseline

PRIVATE TASTE PROFILE:
- Separate storage from personality and public profile data
- 90-day rolling retention window
- Deletable via Trust Hub
- Never shown to other users
- Never used in explanations
- Never used for personality inference

NEVER SHOW: compatibility percentages, numeric scores, soulmate/destiny language, negative incompatibility indicators

SKILL 6 — kesher-why-this-match
Provenance-Labeled Match Explanations
════════════════════════════════════════════════════════════════════

SIGNAL ALLOWLIST (only these may appear in explanations):
- Shared values (both profiles, both public)
- Relationship intent (both profiles, both public)
- Observance/lifestyle (both users chose to display)
- Location proximity (coarse only — neighborhood, not address)
- Shared interests/hobbies (explicitly listed by both)
- Age/stage alignment (within each other's stated range)
- Personality insight (ONLY if sender shared a card with this recipient)

SIGNAL EXCLUSION LIST (never use):
- Private personality scores
- Private taste profile
- Behavioral patterns (swipes, dwell time)
- Inferred traits
- Attractiveness/desirability ranking
- Other users' data about this person

SOURCE CHIPS (required on every explanation element):
- "From your profile" (blue chip)
- "From their profile" (green chip)
- "Shared with you" (purple chip — personality card)
- "You both listed" (gold chip — mutual overlap)

SAFE FORMULATIONS:
✓ "You both value [shared value]."
✓ "You're both looking for [shared intent]."
✓ "[Name] shared that they tend toward [trait from shared card]."

PROHIBITED FORMULATIONS:
✗ "You're X% compatible."
✗ "Our AI thinks you'd be great together."
✗ "Based on your personality, this is a strong match."
✗ "Better match than others in your queue."

MAX 3 explanation items per match. Generic fallback if no overlap found.

SKILL 8 — kesher-psychometric-validation
Validation Pipeline Before Production Launch
════════════════════════════════════════════════════════════════════

VALIDATION PHASES (in order, each has go/no-go gate):
1. Adaptation lab → cognitive interviews, forward/back translation, Hebrew + English
2. Psychometric alpha study → omega ≥ .80 (domains), ≥ .70 (facets), ESEM structure
3. Test-retest stability → domain retest ≥ .75, mean drift < .20 SD
4. Response quality audit → <10% flagged sessions in live traffic
5. Measurement invariance → ΔCFI < .01, ΔRMSEA < .015 across groups
6. Incremental validity → personality adds replicated value over explicit baseline
7. Harm/creepiness testing → no harm measure exceeds control by >0.3 SD

CURRENT STATUS: Phases 1–4 implementable now. Phases 5–7 blocked until traffic and data available.

PERSONALITY IN RANKING IS BLOCKED until all 7 phases pass.

USE ESEM/BIFACTOR (not naive CFA) — personality inventories have systematic cross-loadings that CFA forces to zero, producing misleading results.

SKILL 9 — kesher-compatibility-reflection
Bilateral Consent Reflection Lenses
════════════════════════════════════════════════════════════════════

PREREQUISITES: Both users have shared at least a basic card with each other AND both explicitly consented to mutual reflection.

REFLECTION LENSES (max 3 per pair):
1. Values Alignment: shared priorities from explicit profile values + shared cards
2. Communication: extraversion/agreeableness dynamics, conversation style
3. Friction Forecast: potential tension points, framed as "things to notice," not dealbreakers
4. Growth Edge: differences that might create positive challenge

SAFE LANGUAGE: "might," "could," "sometimes," "tends to," "may find that"

PROHIBITED:
- Compatibility scores or percentages
- Soulmate, destiny, perfect-match, meant-to-be language
- Prediction of relationship success/failure
- Advice on whether to pursue the connection
- One-sided advice ("you should adapt to their style")
- Comparison to population norms

REVOCATION: Either user revoking their shared card → reflection invalidated for both.
Either user revoking mutual consent → reflection disappears for both.

ROUTE THROUGH: Vertex AI (sensitive dyadic data)

SKILL 10 — kesher-dark-pattern-audit
EU Dark Pattern Taxonomy & Comprehension Standards
════════════════════════════════════════════════════════════════════

SIX PROHIBITED DARK PATTERN CATEGORIES:

1. OVERLOADING: >150 words per consent screen; >3 primary actions; legal jargon without explanation
2. SKIPPING: consent not full-screen; sensitive toggles default ON; key info below fold
3. STIRRING: confirm-shaming; giant accept / tiny decline; FOMO/social proof urgency
4. OBSTRUCTING: revoke harder than grant; delete buried >2 taps; extra friction to confirm revocation
5. FICKLE: inconsistent terminology; current consent state not visible; scope unclear
6. LEFT IN THE DARK: no AI disclosure; recipient not specified; no retention info; silent feature changes

COMPREHENSION STANDARDS:
- ≥80% of users can correctly answer: "What did you just agree to?", "Who can see this?",
  "How do you undo this?", "What happens if you say no?", "Is this permanent?"
- If any question fails: revise consent copy and retest

REGRET MEASURES (24-hour follow-up after sharing):
- ≥90% recall the action
- ≥85% would make the same choice again
- <15% surprised by what was shared
- Mean pressure score ≤2.0/5.0

PREMIUM BOUNDARY RULES:
- Core matching never degraded for free users
- Reflection available to all who complete assessment (not premium-gated)
- No "compatibility score unlock" for premium (scores are prohibited entirely)
- No "personality boost" that increases visibility
